# Оптимизация процесса сборки Docker образов

## Проблема

При каждом деплое использовалась команда `docker compose build --no-cache`, что приводило к:
- Созданию нового образа при каждом деплое
- Накоплению старых образов на диске
- Медленной сборке (без использования кэша)
- Переполнению диска

## Решение

### Использование кэша по умолчанию

Теперь по умолчанию используется кэш Docker, что:
- ✅ Ускоряет сборку (пересобираются только изменённые слои)
- ✅ Экономит место на диске (старые образы не накапливаются так быстро)
- ✅ Ускоряет деплой

### Команды

```bash
# Обычная сборка (с кэшем) - используется по умолчанию
./dev.sh prod-build
./dev.sh dev-build
./dev.sh build

# Полная пересборка (без кэша) - только при необходимости
./dev.sh prod-build --no-cache
./dev.sh dev-build --no-cache
./dev.sh build --no-cache
```

## Когда использовать `--no-cache`

### ✅ Обязательно использовать `--no-cache`:

1. **Изменения в Dockerfile**
   - Добавление/удаление системных пакетов
   - Изменение порядка команд
   - Изменение базового образа

2. **Обновление системных зависимостей**
   - Обновление PHP версии или расширений
   - Обновление системных библиотек
   - Изменения в конфигурации системы

3. **Изменения в зависимостях проекта**
   - Обновление `composer.json` / `composer.lock`
   - Обновление `package.json` / `package-lock.json`
   - Изменения в зависимостях, которые влияют на образ

4. **Проблемы с кэшем**
   - Неожиданное поведение приложения
   - Подозрение на устаревший кэш
   - Ошибки при сборке

5. **Профилактика**
   - Раз в месяц для полной пересборки
   - После крупных обновлений системы

### ❌ НЕ нужно использовать `--no-cache`:

1. **Обычные изменения кода**
   - Изменения в PHP файлах
   - Изменения в Blade шаблонах
   - Изменения в JavaScript/CSS
   - Изменения в конфигурации Laravel

2. **Рутинные деплои**
   - Большинство деплоев можно делать с кэшем
   - Это ускорит процесс и сэкономит место

## Автоматическая очистка

При запуске `prod-up` автоматически выполняется очистка старых образов:

```bash
./dev.sh prod-up
# Автоматически удаляет образы старше 3 дней
```

Это предотвращает накопление старых версий на диске.

## Оптимизация Dockerfile

Для максимальной эффективности кэша, Dockerfile должен быть структурирован правильно:

```dockerfile
# 1. Сначала зависимости (меняются редко)
COPY composer.json composer.lock ./
RUN composer install --no-dev --optimize-autoloader

# 2. Потом код (меняется часто)
COPY . .
```

Таким образом, при изменении кода не будет пересобираться слой с зависимостями.

## Мониторинг использования кэша

Проверить использование кэша можно:

```bash
# Размер образов
docker images --format "table {{.Repository}}\t{{.Tag}}\t{{.Size}}"

# Использование диска Docker
docker system df

# Детальная информация
docker system df -v
```

## Рекомендации

1. **Используйте кэш по умолчанию** - это ускорит деплой и сэкономит место
2. **Используйте `--no-cache` только при необходимости** - когда меняются зависимости или Dockerfile
3. **Мониторьте размер образов** - регулярно проверяйте использование диска
4. **Настройте автоматическую очистку** - через cron или в процессе деплоя

## Примеры workflow

### Обычный деплой (изменения в коде)

```bash
# 1. Сборка с кэшем (быстро)
./dev.sh prod-build

# 2. Запуск (автоматическая очистка старых образов)
./dev.sh prod-up
```

### Деплой после обновления зависимостей

```bash
# 1. Полная пересборка (без кэша)
./dev.sh prod-build --no-cache

# 2. Запуск
./dev.sh prod-up
```

### Деплой после изменений в Dockerfile

```bash
# 1. Обязательно без кэша
./dev.sh prod-build --no-cache

# 2. Запуск
./dev.sh prod-up
```

## Результат

После оптимизации:
- ✅ Деплой стал быстрее (используется кэш)
- ✅ Меньше нагрузка на диск (старые образы удаляются автоматически)
- ✅ Меньше проблем с переполнением диска
- ✅ Более эффективное использование ресурсов
