# Оптимизация процесса деплоя

## Проблема

Деплой занимал 10+ минут, потому что каждый раз Docker собирал образы с нуля, даже при обычных изменениях кода.

## Решение

### 1. Умная сборка с кэшем

Workflow теперь автоматически определяет, нужна ли полная пересборка:

- **С кэшем** (по умолчанию) - когда изменяется только код приложения
- **Без кэша** (`--no-cache`) - только при изменениях в:
    - `Dockerfile`
    - `docker/`
    - `docker-compose.production.yml`
    - `composer.json` / `composer.lock`
    - `package.json` / `package-lock.json`
    - `vite.config.js`

### 2. Автоматическая очистка

После каждого деплоя автоматически удаляются образы старше 3 дней, что предотвращает переполнение диска.

### 3. Использование новой версии Docker Compose

Переход с `docker-compose` (старая версия) на `docker compose` (новая версия, встроенная в Docker).

## Как это работает

```bash
# Workflow автоматически проверяет изменения
git diff OLD_SHA NEW_SHA --name-only | grep -E "(Dockerfile|docker/|docker-compose\.production\.yml|composer\.(json|lock)|package(-lock)?\.json|vite\.config\.js)"

# Если изменения найдены → полная пересборка
docker compose build --no-cache

# Если изменений нет → сборка с кэшем (быстро!)
docker compose build

# После деплоя → migration/cache/health и очистка старых образов
docker image prune -a -f --filter "until=72h"
```

## Результат

- ✅ **Деплой стал быстрее** - обычно 2-3 минуты вместо 10+
- ✅ **Меньше нагрузка на диск** - старые образы удаляются автоматически
- ✅ **Умная сборка** - полная пересборка только при необходимости
- ✅ **Экономия ресурсов** - используется кэш Docker

## Когда будет полная пересборка

Полная пересборка (`--no-cache`) выполняется автоматически, если в коммите изменены:

- `Dockerfile`
- `docker/`
- `docker-compose.production.yml`
- `composer.json`
- `composer.lock`
- `package.json`
- `package-lock.json`
- `vite.config.js`

Для всех остальных изменений (PHP код, Blade, JS, CSS и т.д.) используется кэш.

## Production cutover

Production backend деплоится из ветки `main` в текущий dev-слот сервера (`/opt/vedma-dev`) и по умолчанию публикуется на порту `8000`, чтобы основной домен продолжил попадать в backend без ручной смены upstream.

Перед остановкой dev workflow создает backup базы и `storage/app`, затем поднимает `docker-compose.production.yml`, выполняет `migrate --force`, Laravel cache optimize и проверку `/health`.

Подробный порядок переключения и rollback описан в [PRODUCTION_RUNBOOK.md](./PRODUCTION_RUNBOOK.md).

## Ручная полная пересборка

Если нужно принудительно сделать полную пересборку (например, при проблемах), можно:

1. **Временно изменить Dockerfile** (добавить комментарий)
2. **Или изменить composer.json** (добавить/удалить комментарий)
3. **Или использовать специальный коммит** с сообщением `[rebuild]`

## Мониторинг

В логах деплоя вы увидите:

```
Изменений в Dockerfile нет, используем кэш...
```

или

```
Обнаружены изменения в Dockerfile или composer.json, делаем полную пересборку...
```

## Дополнительные оптимизации

### Оптимизация Dockerfile

Для максимальной эффективности кэша, Dockerfile структурирован так:

```dockerfile
# 1. Сначала зависимости (меняются редко) - кэшируется
COPY composer.json composer.lock ./
RUN composer install --no-dev --optimize-autoloader

# 2. Потом код (меняется часто) - пересобирается только при изменениях
COPY . .
```

Таким образом, при изменении кода не пересобирается слой с зависимостями.

## Troubleshooting

### Деплой всё ещё медленный

1. Проверьте, что на сервере установлена новая версия Docker (с `docker compose`)
2. Проверьте использование диска: `df -h`
3. Проверьте кэш Docker: `docker system df`
4. При необходимости очистите вручную: `./docker/cleanup.sh`

### Нужна принудительная пересборка

Если нужно принудительно пересобрать всё:

1. Временно измените `Dockerfile` (добавьте комментарий)
2. Закоммитьте и запушьте
3. После деплоя верните изменения обратно

### Проблемы с кэшем

Если подозреваете проблемы с кэшем:

1. Проверьте логи деплоя
2. Убедитесь, что на сервере достаточно места
3. Проверьте версию Docker: `docker --version` и `docker compose version`
